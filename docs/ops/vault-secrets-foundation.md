# Vault Secrets Foundation Runbook

## Scope

Story `0-8-vault-secrets-foundation` baseline for centralized secret management.

For staging/prod separation and operator-managed external Vault flows, continue with `docs/ops/vault-external-operations.md`.

Story `12.4` DMZ privileged operator access is not satisfied by this local bootstrap baseline.

Non-local `issue`, `inspect`, and `revoke` flows must follow `docs/ops/vault-external-operations.md` and `docs/ops/dmz-admin-access.md`.

For a low-cost non-local dev server that still keeps persistent `raft` storage in Docker, continue with `docs/ops/vault-external-dev-server.md`.

- Vault engine: `KV v2` at `secret/`
- CI auth: `GitHub Actions OIDC` to `auth/jwt`
- Runtime auth: `AppRole` with short-lived tokens
- Critical secret used in this baseline: internal boundary secret shared by core backend services

## Secret Path Conventions

| Domain | Path | Notes |
| --- | --- | --- |
| Shared core services | `secret/fix/shared/core-services/internal-secret` | Internal boundary secret (`X-Internal-Secret`) |
| Channel config | `secret/fix/channel/<env>/<key>` | Reserved for future story expansion |
| Corebank config | `secret/fix/corebank/<env>/<key>` | Reserved for future story expansion |
| FEP gateway/simulator | `secret/fix/fep/<service>/<env>/<key>` | Reserved for future story expansion |

Naming rule: `<domain>/<service>/<env>/<secret-key>` with lowercase `kebab-case`.

## Role and Policy Matrix (Least Privilege)

| Identity | Auth Method | Vault Role | Policy | Allowed Capabilities |
| --- | --- | --- | --- | --- |
| Docs publish workflow | GitHub OIDC JWT | `ci-docs-publish` | `ci-docs-publish` | `read` only on `secret/fix/shared/core-services/internal-secret` |
| Runtime services (corebank/fep/channel) | AppRole | `runtime-core-services` | `runtime-internal-secret` | `read` only on `secret/fix/shared/core-services/internal-secret` |

Policy files are stored at:

- `docker/vault/policies/ci-docs-publish.hcl`
- `docker/vault/policies/runtime-internal-secret.hcl`

## Bootstrap and Runtime Flow

1. Fill required variables in `.env` (from `.env.example`):
   - `VAULT_DEV_ROOT_TOKEN_ID`
   - `INTERNAL_SECRET_BOOTSTRAP`
2. Start Vault baseline for local/dev only:

```bash
docker compose -f docker-compose.vault.yml up -d vault vault-init
```

3. Read runtime AppRole credentials generated by bootstrap:

```bash
export VAULT_RUNTIME_ROLE_ID="$(docker exec vault cat /vault/file/runtime-role-id)"
export VAULT_RUNTIME_SECRET_ID="$(docker exec vault cat /vault/file/runtime-secret-id)"
export VAULT_ROTATION_ROLE_ID="$(docker exec vault cat /vault/file/rotation-role-id)"
export VAULT_ROTATION_SECRET_ID="$(docker exec vault cat /vault/file/rotation-secret-id)"
```

4. Resolve runtime `INTERNAL_SECRET` from Vault path:

```bash
export INTERNAL_SECRET="$(
  VAULT_ADDR=http://localhost:8200 \
  VAULT_RUNTIME_ROLE_ID="$VAULT_RUNTIME_ROLE_ID" \
  VAULT_RUNTIME_SECRET_ID="$VAULT_RUNTIME_SECRET_ID" \
  ./docker/vault/scripts/read-internal-secret.sh
)"
```

5. Start application stack:

```bash
docker compose up -d
```

Runtime behavior is **fail-fast**: compose requires `INTERNAL_SECRET` and aborts startup when not provided.

Local bootstrap can optionally use `COMPOSE_PROFILES=local-vault`. Non-local environments must not enable that profile.

## CI Integration (GitHub Actions OIDC)

`docs-publish.yml` uses:

- `permissions.id-token: write`
- OIDC token retrieval from GitHub runner
- Vault login command:

```bash
vault write -field=token auth/jwt/login role=ci-docs-publish jwt="$OIDC_JWT"
```

When `VAULT_ADDR` secret is missing, CI enters controlled **degraded mode** (`VAULT_SECRET_MODE=degraded`) and skips Vault secret resolution.

## Rotation drill

Run one critical-secret rotation drill:

```bash
VAULT_ADDR=http://localhost:8200 \
VAULT_RUNTIME_ROLE_ID="$VAULT_ROTATION_ROLE_ID" \
VAULT_RUNTIME_SECRET_ID="$VAULT_ROTATION_SECRET_ID" \
EVIDENCE_FILE=docs/ops/evidence/vault-rotation-drill.log \
./docker/vault/scripts/rotate-internal-secret.sh
```

Note: runtime service AppRole (`runtime-core-services`) is read-only by design. Rotation must use the dedicated rotation role (`ops-rotate-internal-secret`) or approved operator token.

Expected output: `rotation_ok secret/fix/shared/core-services/internal-secret <timestamp>`

After rotation, re-read runtime secret and restart dependent services only if needed. No user-visible outage target: no failed health endpoint for the edge-exposed `channel-service`.

## Audit Verification

Enable and verify file audit:

```bash
vault audit list
```

Default path from bootstrap: `${VAULT_AUDIT_LOG_PATH}` (usually `/vault/file/audit.log`).

Check read/write traces include actor and path metadata:

```bash
docker compose exec -T vault sh -lc "tail -n 20 /vault/file/audit.log"
```

Retention expectation for this baseline: keep audit evidence for at least 90 days in CI artifacts or ops archive.

## Vault-Unreachable Chaos Check

Run:

```bash
VAULT_FAILURE_MODE=fail-fast \
SERVICE_HEALTH_URL=http://localhost:8080/actuator/health \
./docker/vault/scripts/check-vault-unreachable.sh
```

Verification:

- secret read fails within timeout (`VAULT_READ_TIMEOUT_SECONDS`, default `5`)
- service behavior matches documented fallback:
  - **fail-fast**: service unavailable or health probe failure
  - **degraded mode**: endpoint responds with explicit degraded status signaling
