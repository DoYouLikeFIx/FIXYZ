## Story Pattern Synthesis

### Common Struggles

- Boundary contracts, error taxonomy, and idempotency had to be treated as one system.
- Drift between canonical planning docs and supplemental implementation notes remained a repeated friction point.
- Cross-client visible error handling depended on backend contract stability.

### Recurring Review Feedback Themes

- Review feedback favored explicit retry, status-query, and escalation boundaries over implicit resilience behavior.
- Contract-test depth was treated as the main defense against silent integration drift.
- Parity between backend error codes and FE/MOB messaging remained a recurring requirement.

### Breakthrough Moments

- The epic established a clearer base for later execution and resilience work.
- Added canonical success fixtures for `PARTIALLY_FILLED`, `PENDING`, `REJECTED`, `CANCELED`, and `MALFORMED` status responses and parameterized the status success-path mapping assertions across the full documented matrix.
- Tightened FE/MOB order-account handling to canonical numeric IDs only, gated the order boundary for users without a linked order account, and upgraded FE/MOB E2E harnesses to assert request-contract behavior instead of preview-only rendering.

### Technical Debt and Risks

- If future execution or resilience work bypasses the agreed FEP contract, later recovery flows will become inconsistent.
- Supplemental archival docs can still confuse ownership unless canonical mapping remains explicit.
- Error UX parity can regress quickly if downstream teams re-interpret external failure semantics locally.
