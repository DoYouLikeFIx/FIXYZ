## What Went Well

- The FEP boundary is now deterministic across submit, status, error, and idempotency flows instead of being defined only at the happy-path API level.
- Error normalization now travels end to end from FEP-facing services to channel-facing UX, including `userMessageKey` and `operatorCode` propagation.
- Status requery semantics were hardened with explicit retry and escalation metadata, which gives later recovery work a stable contract to build on.
- The contract test suite moved to canonical reusable fixtures plus OpenAPI compatibility checks, which materially reduced silent drift risk.
- FE and MOB external-order failure UX regained parity through a shared contract and matching automated coverage.
- Epic 3 closeout is now reflected consistently in the sprint tracker and story artifacts, removing the previous mismatch between implementation reality and artifact status.

## Challenges And Lessons

- Historical numbering drift between supplemental epic artifacts and canonical `epics.md` created avoidable confusion during closeout, especially around resilience-related scope.
- Story metadata lagged behind implementation status in several Epic 3 artifacts, which meant the final closeout required a normalization pass across story files and `sprint-status.yaml`.
- Some implementation artifacts still contain encoding noise or low-portability references, which weakens the usefulness of otherwise solid technical documentation.
- Epic 3 implementation proved several resilience-adjacent capabilities, but the canonical roadmap still assigns broader timeout, chaos, and recovery orchestration scope to later epics. Those boundaries must stay explicit.
- Cross-repo and cross-client work remains vulnerable to drift when FE, MOB, and backend changes do not get revalidated together. Story `3.6` showed that parity can regress unless the shared contract stays central.
