## Story Pattern Synthesis

### Common Struggles

- Timeout, retry, requery, replay, and degraded UX had to be designed as one system.
- Operator controls and runtime behavior needed to stay traceable to each other.
- Failure-mode confidence required more explicit evidence than a normal feature epic.

### Recurring Review Feedback Themes

- Review feedback emphasized bounded retry behavior, clear circuit-breaker semantics, and explicit escalation states.
- Scenario tests and replay/recovery evidence were treated as first-class deliverables.
- Degraded-operation UX stayed important because resilience is visible to users when things go wrong.

### Breakthrough Moments

- Resilience work now spans backend mechanics and visible degraded UX in a way later epics can reuse.
- Applied adversarial review fixes so fresh `UNKNOWN` execution outcomes enter `REQUERYING`, detached JPA writes reload managed entities, and REQUERYING backlog traversal now uses an `(updatedAt, orderSessionId)` cursor instead of growing `NOT IN (...)` exclusions.
- Implemented the internal replay lane in `corebank-service` and replay result translation from `fep-gateway`, preserving `ESCALATED` on `CORE-001` retry-later failures and finalizing `COMPLETED` / `FAILED` / `CANCELED` outcomes.

### Technical Debt and Risks

- Later orchestration work can mask resilience regressions unless the failure-path suite remains active.
- Manual recovery APIs become risky when documentation and runtime behavior diverge.
- No prior continuity record means some improvement commitments still need to be institutionalized now.
