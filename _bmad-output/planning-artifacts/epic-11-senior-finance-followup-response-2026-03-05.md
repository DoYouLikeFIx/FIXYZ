# 2차 피드백 반영 보고서

## 1. 보고 목적

추가 피드백을 기준으로, 설계 고도화 이후 단계에서 무엇을 먼저 구현하고 어떤 순서로 검증·운영까지 확장할지 업무 관점에서 설명하기 위해 작성했습니다.

---


## 2. 전체 요약

기존 설계는 구조적으로 안정적이었고, 변경 이유를 설명할 기본 틀이 갖춰져 있었습니다.
이번 반영에서는 그 기반 위에, 먼저 실제로 동작하는 핵심 기능을 보여주기 위한 구현 순서, 검증 시나리오, 운영 화면, 배포 준비, 외부 구간/내부 구간 인증 흐름 설명을 구체화했습니다.

---

## 3. 피드백 항목별 반영 내용

## 3.1 구현 우선순위(핵심 거래 흐름 우선)

### 이미 잘 되어 있던 부분

- 주문, 체결, 잔고, 복구, 운영 통제까지 기능 범위가 정의되어 있었습니다.
- 작업 단위가 나뉘어 있어 구현 시작 기반이 마련되어 있었습니다.

### 이번에 보강한 부분

- 구현 순서를 2단계로 명확히 정했습니다.
  - 1단계: 주문 접수 → 체결 → 잔고 반영까지 먼저 동작하도록 구현
  - 2단계: 장애 처리, 운영 통제, 고급 복구 기능을 순차 확장
- 설계 전체를 한 번에 구현하지 않고, 먼저 핵심 기능의 실제 동작을 증명하는 방식으로 정리했습니다.

### 개선 효과

- 일정과 구현 위험을 줄이면서, 포트폴리오에서 가장 중요한 "동작하는 핵심 기능"을 우선 보여줄 수 있습니다.

---

## 3.2 테스트 시나리오 문서 포함(최종 통합 검증 단계 중심)

### 이미 잘 되어 있던 부분

- 최종 통합 검증 단계에 자동 검증, 성능 검증, 장애 복구 점검, 배포 전 점검 항목이 정의되어 있었습니다.
- 기본 기능 테스트와 테스트 코드 연결표가 이미 문서화되어 있었습니다.

### 이번에 보강한 부분

- 최종 통합 검증 단계 기준 테스트 시나리오를 아래와 같이 정리했습니다.

| 영역 | 테스트 시나리오 | 확인하려는 내용 | 통과 기준 |
|---|---|---|---|
| 핵심 기능 검증 | 주문 접수→체결→완료 정상 처리 | 기본 주문 흐름 안정성 | 전체 흐름 성공, 상태 전이 정상 |
| 핵심 기능 검증 | 동일 종목 동시 매수/매도 100건 발생 | 동시 처리 시 보유 수량 정합성 | 최종 보유수량·평균단가·주문상태 불일치 0건 |
| 핵심 기능 검증 | 인증번호 오류 입력 시 주문 실행 차단 | 주문 전 본인 확인 통제 | 주문 차단, 실패 상태 일관성 유지 |
| 핵심 기능 검증 | 같은 주문번호 재요청 | 중복 처리 방지 | 중복 체결/중복 잔고 반영 0건 |
| 핵심 기능 검증 | 외부 연계 지연 반복 상황 | 외부 장애 보호장치 동작 | 즉시 실패 응답, 보유수량/예수금 무변동 |
| 핵심 기능 검증 | 로그아웃 후 동일 사용자 재요청 | 세션 보안 | 후속 요청 인증 거절 |
| 핵심 기능 검증 | 누적 매수·매도와 보유수량 대조 | 원장 정합성 | 종목별 수량 불일치 0건 |
| 핵심 기능 검증 | 시세 수신 3초 초과 지연 상태 주문 요청 | 지연 시세 기반 주문 차단 | 주문 거절 + 지연 사유 기록 |
| 핵심 기능 검증 | 부분 체결 후 남은 수량 취소 | 체결/취소 혼합 상황 손익 정확성 | 실현손익·평가손익·잔량 계산 오차 0 |
| 보안 경계 검증 | 내부 전용 주소를 인증 없이 접근 | 외부/내부 경계 통제 | 접근 차단(인증 또는 권한 오류 응답) |
| 성능 검증 | 응답시간 기준 충족 여부 확인 | 사용자 체감 지연 통제 | 응답시간 목표(상위 95% 구간) 충족 |
| 성능 검증 | 성능 기준 초과 시 배포 차단 | 품질 기준 강제 | 기준 초과 시 배포 자동 차단 |
| 성능 검증 | 반복 측정 결과 편차 확인 | 성능 안정성 | 허용 편차 초과 시 경고 또는 실패 처리 |
| 복구 점검 | 외부기관 응답 모의 도구(FEP Simulator)로 지연/실패 반복 주입 | 장애 확산 방지 | 차단 상태 진입 확인 |
| 복구 점검 | 외부기관 응답 모의 도구를 정상 응답 모드로 전환 | 서비스 복원력 | 정상 상태 복귀 후 주문 재개 |
| 복구 점검 | 미해결 주문 재조회/재처리 | 복구 수렴성 | 정상 수렴 또는 관리자 조치 상태로 명확 전환 |
| 배포 전 점검 | 통합 환경 기동 직후 점검 | 배포 직후 안정성 | 정해진 시간 내 핵심 상태 정상 |
| 배포 전 점검 | 필수 서비스 주소 응답 확인 | 필수 기능 가용성 | 필수 주소 응답 성공 |
| 배포 전 점검 | 롤백 절차 리허설 | 장애 시 복구 가능성 | 롤백 절차 실행 가능 + 증적 확보 |
| 관측 점검 | 상태 수집 도구에서 서비스 상태 확인 | 운영 상태 가시성 | 모니터링 대상 상태 정상 |
| 관측 점검 | 운영 화면 접속 및 기본 패널 확인 | 운영 즉시 대응성 | 운영 화면 접속 + 핵심 패널 표시 확인 |
| 웹 출시 점검 | 로그인~주문~결과 핵심 여정 점검 | 웹 사용자 핵심 흐름 | 핵심 여정 전부 통과 |
| 모바일 출시 점검 | 단말/시뮬레이터 핵심 흐름 점검 | 모바일 호환성/안정성 | 대상 기기 조건에서 핵심 흐름 통과 |

- 복구·장애 시나리오는 외부기관 응답 모의 도구를 활용해 같은 결과를 반복 검증하도록 명시했습니다.

### 개선 효과

- 테스트 결과 보고 시 시나리오 이름만으로도 검증 범위와 목적을 빠르게 추적할 수 있습니다.

## 3.3 운영 대시보드(최소 기능 화면)

### 이미 잘 되어 있던 부분

- 운영 통제와 상태 관리 개념이 설계에 반영되어 있었습니다.
- 체결/미체결/시세 상태 데이터를 수집할 구조가 마련되어 있었습니다.

### 이번에 보강한 부분

- 운영 화면의 최소 범위를 명확히 정의했습니다.
  - 실시간 체결 현황
  - 미체결 건수
  - 시세 수신 상태
- 기능 화면과 별도로, 운영 관점에서 시스템 상태를 한눈에 확인하는 화면을 명세에 포함했습니다.

### 개선 효과

- 기능 구현뿐 아니라 운영 준비 수준까지 함께 보여줄 수 있어 신뢰도를 높일 수 있습니다.

---

## 3.4 설계에서 실행 단계로의 전환

### 이미 잘 되어 있던 부분

- 설계 품질과 문서 정합성이 확보되어 있었습니다.
- 확장 가능한 구조 기반이 마련되어 있었습니다.

### 이번에 보강한 부분

- 실행 전략을 문서 중심에서 결과 중심으로 전환했습니다.
  - 바텀업 기반으로 각 레이어별로 핵심 기능이 동작하는지 구현 하는 방식으로 정리했습니다.
  - 이후 통합기능을 구현하여 통제/복구/운영 기능을 순차 확장하는 방식으로 정리했습니다.

### 개선 효과

- 설계와 구현 사이 간극을 줄이고, 리뷰 시점마다 실제 동작 결과를 제시할 수 있습니다.

---

## 3.5 배포 자동화, 형상관리, 공통 모듈 관리

### 이미 잘 되어 있던 부분

- 모듈 분리와 공통 구성요소 활용 기반이 있었습니다.
- 자동 빌드/배포로 확장 가능한 구조가 마련되어 있었습니다.

### 이번에 보강한 부분

- 핵심 기능 안정화 이후 자동 배포(CD)를 단계적으로 도입하도록 계획했습니다.
- 형상관리 운영 원칙(브랜치, 변경 검토, 병합 기준)과 공통 모듈 버전 관리 개념을 필수 항목으로 명시했습니다.
- 결과물이 반복 가능하게 배포·공유되도록 저장소 운영 관점을 문서에 반영했습니다.

### 개선 효과

- 단발성 시연을 넘어, 지속 가능한 개발·배포 체계로 확장할 기준이 명확해졌습니다.

---

## 3.6 외부 접속 구간/내부 처리 구간 및 쿠키-세션 흐름

### 이미 잘 되어 있던 부분

- 사용자 접점 영역과 업무 처리 영역을 분리하는 방향이 반영되어 있었습니다.
- 세션 기반 인증·보안 정책의 기본 원칙이 존재했습니다.

### 이번에 보강한 부분

- 영역 구분을 업무 용어로 명확히 정리했습니다.
  - 프론트엔드: 외부 접속 구간
  - 백엔드: 내부 처리 구간
- 인증 흐름을 한 줄로 설명 가능하게 정리했습니다.
  - 사용자 단말 → 쿠키 전달 → 프론트엔드 요청 → 세션 확인 → 백엔드 처리
- 쿠키 보안 속성, 세션 유지/만료, 요청 인증 전달 과정을 하나의 운영 흐름으로 연결했습니다.

### 개선 효과

- 네트워크/보안 질문에 대해 구조적으로 설명할 수 있어, 실무 이해도를 분명하게 전달할 수 있습니다.

---

## 3.7 문서 우선 반영 원칙(코드 변경과 분리)

### 이미 잘 되어 있던 부분

- 설계·계획·시나리오 문서가 분리되어 있어, 코드와 별개로 문서 보강이 가능한 구조였습니다.

### 이번에 보강한 부분

- 이번 라운드는 코드 수정 없이 문서 반영을 먼저 진행한다는 원칙을 명시했습니다.
- 반영 범위를 "프로젝트 마지막 통합 검증 단계"로 고정했습니다.
  - 수용 시나리오 목록과 통과 기준 정리
  - 성능/복구/보안/배포 점검 기준 명시
  - 모니터링 기반 운영 점검 항목 추가

### 개선 효과

- 구현 진행도와 무관하게, 리뷰어가 현재 검증 계획과 운영 준비도를 문서만으로 확인할 수 있습니다.

---

## 5. 결론

특히 다음 두 가지가 명확해졌습니다.

1. 다음 단계의 핵심은 핵심 거래 흐름이 실제로 동작하는 상태를 먼저 구현·검증하는 것입니다.
2. 이후 테스트 시나리오, 운영 화면, 자동 배포, 인증·네트워크 설명을 순차적으로 연결해야 실무형 완성도를 높일 수 있습니다.
